Part I

核心概念

什么是 Vibe Coding

一种全新的编程范式：你不再逐行写代码，而是用自然语言描述意图，让 AI 代理生成、调试、迭代整个系统。你的角色从"写代码的人"转变为"导演"——把控方向、审视产出、做关键决策。

这不意味着降低门槛，恰恰相反：它对需求表达的精确性、审美判断力、系统思维提出了更高要求。

SSpec（规格说明）

写给 AI 的"施工蓝图"：目标、功能需求、约束条件、视觉与交互意图、验收标准。Spec 的质量直接决定产出的质量。

类比：Spec 之于 Vibe Coding，就像实验方案之于科研——写得好，才能复现、迭代、交流。

PPlan Mode（规划模式）

Claude Code 的只读分析模式，AI 只探索代码、分析结构、提出方案，不做任何修改。通过 Shift+Tab 或 claude --permission-mode plan 启用。

先用 Plan Mode 对齐方案，再进入实现。先想清楚再动手。

AAgentic Loop（自主循环）

Claude Code 的核心工作方式：分析 → 行动 → 验证 → 修正，自动迭代直到完成。你可以在任何时候中断并纠偏方向。

  • 1. 委托，不是口述
  • 2. 先规划，后实现
  • 3. 持续校准
  • 4. 审美是功能
  • 5. 迭代 Spec
  • 6. 善用生态

Part IV

综合考核：Research Field Navigator

总述

构建一个研究领域导航器：输入你自己博士课题所在的研究领域，系统自动完成从数据采集、文献图谱、趋势分析、争议辩论到研究提案的完整科研情报链。

由 7 个模块组成，你需要完成全部模块，但每个模块的深度可以有所取舍——建议选择 2-3 个模块重点打磨，其余保证基本可用。

关键约束：必须以你自己的博士研究领域作为输入，所有模块的产出必须围绕你的真实研究方向。
1 数据采集→ 2 文献图谱→ 3 趋势统计→ 4 研究提案→ 5 争议分析→ 6 综合界面→ 7 反思
1 数据采集管线 10%

构建端到端的论文数据采集管线。

  • 从至少两个不同的公开数据源获取你研究领域近 1-2 年的论文元数据
  • 处理好分页、限流、数据源格式差异
  • 清洗脏数据，跨数据源去重，统一持久化存储
  • 支持增量更新（不重复拉取已有数据）
  • 总量不少于 200 篇论文
数据工程 多源整合 边界条件处理
维度分值AC
多源整合402+ 数据源适配良好；跨源去重合理单源跑通但多源有问题
工程质量30增量更新正确；有日志和错误恢复；200+ 篇完整能采集但无增量或数据量不足
边界处理30分页/限流/网络异常/缺失字段均有处理主流程通但异常会崩溃
仅单一数据源且无增量更新，上限 C。数据量不足 100 篇，上限 D。
2 文献图谱可视化 15%

基于采集数据，构建交互式引用关系图谱。

  • 从种子论文出发，递归展开引用关系至 2 层深度，200+ 节点不崩溃
  • 自动检测关键节点（高被引枢纽、跨子领域桥梁）
  • 视觉编码传达信息层级，不依赖图例即可直觉阅读
  • 交互探索：点击查看详情与 AI 摘要，按时间/主题/影响力筛选
可视化品质是本模块的核心评判标准——这是审美能力的重点考察。
图算法 聚类 "自解释"可视化
维度分值AC
数据与图构建202 层正确获取；200+ 不崩溃；关键节点检测正确数据少或有重复；无关键节点
可视化品质50视觉编码自然传达层级；聚类可辨；无需图例；大规模可读节点无差异化或大规模时混乱
交互与洞察30交互流畅；研究脉络识别有洞察基本交互但筛选或脉络缺失
纯静态图片且节点同大小同色，可视化品质记 0。
3 趋势分析与统计 10%

对采集数据进行领域级别的定量分析。

  • 热门关键词趋势变化（时间序列）、活跃作者/机构、新兴子方向
  • 按主题自动聚类，可视化各聚类规模和增长速度
  • 关键趋势提供统计支撑，附带自然语言解读——准确、具体、不故弄玄虚
数据分析 统计素养 专业知识"翻译"为人话
维度分值AC
分析深度35时间序列趋势；有意义的聚类；识别新兴方向；有统计检验基础统计但无聚类或检验
统计正确性30方法合理；前提有检查；措辞与证据强度匹配有数字但方法不当或过度解读
自然语言解读35准确具体，对非专业人士友好但不失准确过于技术化或过于简化
统计方法明显错误且无说明，统计正确性记 0。
4 研究空白与提案 15%

基于前三模块数据，系统性发现研究空白并生成提案。

  • 通过主题建模、引用网络分析、方法-问题交叉等手段识别空白，每个空白必须有数据证据
  • 对每个空白生成结构化研究提案，至少 3 个
  • 设计可解释的 novelty scoring 框架进行自评
  • 每个提案链接到支撑论文，形成可追溯证据链
每个提案应精炼到一页以内，读起来像正式的研究 proposal。
科研方法论 学术写作精炼与说服力
维度分值AC
空白识别方法30有系统方法论；每个空白有数据证据链缺乏系统性或证据薄弱
提案质量40结构清晰精炼有说服力；一页以内完整；有证据链有结构但冗长或空洞
Novelty 评分30多维度、定义明确、可解释可复现维度单一或定义模糊
提案像 AI 默认输出（泛泛而谈），提案质量记 0。空白无数据支撑，空白识别记 0。
5 争议问题多视角分析 15%

选择你研究领域中一个有争议的问题，进行多视角深度分析。

  • 拆解为可辩论的子议题
  • 至少 3 个差异化 AI 角色，基于真实文献多轮交锋
  • 综述报告：有论证节奏、论据强弱区分、共识与分歧标注
  • 每个关键论断标注证据来源和强度（强实证 / 间接推断 / 理论猜想）
报告应读起来像高质量的科技评论，而非 AI bullet point 罗列。
多角色系统 论证逻辑 "AI 输出→人类可读"策展
维度分值AC
辩论质量35角色真正差异化；多轮交锋能回应对方各说各话
报告表达45有节奏；正反对比清晰；证据分级；结论有分寸bullet point 罗列
设计说明20讨论 prompt 设计对辩论质量的影响仅描述架构
各角色独立生成无交互，系统设计记 0。报告无结构，报告表达记 0。
6 综合报告与界面 20%

将所有模块整合为统一的交互界面和综合报告。

  • 交互界面：浏览图谱、趋势、提案和辩论报告，各模块间有自然导航和交叉引用
  • 综合报告：可导出有叙事结构的领域研究报告（概览→图谱→趋势→空白→争议→建议），排版讲究、图文配合
  • 界面品质：清晰的视觉层次、舒适的阅读体验、协调的配色和 typography
  • CLI 入口：至少支持命令行触发采集和查看关键摘要，终端输出同样有设计感
Web 界面和终端输出的双重审美表达——这是整体审美品味的终极考察。
全栈整合 信息架构 Web + CLI 双重审美
维度分值AC
系统整合20数据流通顺畅；模块间有交叉引用；整体是一个产品模块独立但缺少衔接
Web 界面品质30视觉层次清晰；配色协调；信息密度合理；报告图文配合脱离默认样式但拥挤或层级不清
CLI 体验20色彩有语义；信息分层；进度反馈；错误说人话有 CLI 但输出为裸文本
设计说明30说清架构/视觉/交互决策的"为什么"仅描述"做了什么"
无法运行直接 F。纯默认样式，Web 界面记 0。无 CLI 入口，CLI 维度记 0。
7 反思与 Spec 迭代 15%

对整个构建过程的回顾——这不是代码模块，而是对元认知能力的考察。

  • 提交 Spec 的完整迭代历史（至少 2 个版本），展示每次修改的原因
  • 撰写反思报告（800 字以内），回答：
    ① Spec 如何迭代，什么触发了修改？
    ② 各模块中 AI 在哪些环节超出预期？
    ③ 哪些环节必须人工干预，为什么？
    ④ 如果重做一次，你会改变什么？
反思报告的表达质量本身就是评判标准——精确、克制、有洞察。
Spec 迭代 简洁书面表达 批判性反思
维度分值AC
Spec 迭代质量40至少 2 版；演进清晰；修改原因有记录只有最终版或看不出修改原因
反思报告60800 字以内；四个问题都有具体例子；洞察真实；行文精确克制空泛、缺例子、或严重超字数
反思超 1200 字且无洞察，上限 C。未提交迭代历史，Spec 上限 C。研究领域明显编造，直接 D。

Part V

考核安排

时间安排

总时长 6-8 小时。建议先花 30 分钟通读全部模块，理解整体架构后再从模块 1 开始。

深度分配

选择 2-3 个模块重点打磨，其余保证基本可用。评分综合考虑整体完成度和重点模块深度。

提交物

Spec 原文（含迭代版本）、代码仓库（可运行）、截图/录屏、设计说明（200-300 字）、反思报告（800 字以内）。
